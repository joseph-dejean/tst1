# Dataplex Business User Interface - Backend Implementation Guide

This document explains the architecture, API design, and data flow of the Node.js/Express backend for the Dataplex UI. It serves as a guide for any developer who needs to understand, maintain, or extend the API.

## 1. Architectural Overview

The backend is built using **Node.js** and **Express.js**. Its primary job is to act as a secure, authenticated proxy between the React frontend and Google Cloud APIs (Dataplex, BigQuery, Vertex AI, Data Catalog, Cloud IAM, Firestore).

### Key Technologies:
- **Express.js**: Web server framework.
- **Google Cloud SDKs**: `@google-cloud/dataplex`, `@google-cloud/vertexai`, `@google-cloud/bigquery`, etc.
- **Firestore**: Used for caching relationship graphs and managing approval workflows.
- **Vertex AI (Gemini)**: Used for the "Chat with Table" feature and AI-driven search.

---

## 2. Global Structure & Files

- **`server.js`**: The main entry point. Initializes the server, loads environment variables, initializes GCP clients, and registers all main route handlers.
- **`services/`**: Contains business logic separated by domain.
  - `datasetRelationshipService.js`: Handles relationship discovery (triggering DataScans and parsing Catalog aspects).
  - `accessRequestService.js`: Manages Firestore records for data access requests.
  - `gcpIamService.js`: Executes actual IAM role grants/revocations in Google Cloud.
  - `emailService.js`: Sends notification emails.
  - `adminService.js`: Checks user admin permissions.
- **`middlewares/`**: 
  - `authMiddleware.js`: Validates JWT tokens from the frontend against Google OAuth.
  - `errorHandler.js`: (if present) standardized error handling.

## 3. Core API Endpoints

### 3.1. Dataplex Catalog & Entities
- **GET `/api/v1/search`**: Uses Data Catalog API to search for datasets, tables, and glossaries.
- **GET `/api/v1/ai-search`**: Uses Vertex AI to translate human queries ("show me sales data") into Data Catalog queries.
- **POST `/api/v1/get-entry`**: Retrieves details about a specific Dataplex Entry, passing back all Dataplex Aspects (schema, overview, contacts).

### 3.2. Relationship Discovery (DataScans + Gemini)
- **GET `/api/v1/dataset-relationships`**: Fetches inferred relationships between tables.
  - **How it works**: It checks Firestore cache. If missing, it queries the Dataplex Catalog API for the `dataDocumentation` aspect (generated by Gemini). If the aspect is missing, it automatically triggers a Dataplex DataScan to create it.
  - **File Reference**: Managed primarily in `services/datasetRelationshipService.js`.

### 3.3. Access Management (IAM & Workflows)
- **POST `/api/v1/access-request`**: Submits a request for table/dataset access. Writes to Firestore.
- **POST `/api/v1/access/update`**: Approves/Rejects a request. If approved, `gcpIamService.js` grants the IAM role directly to the user on the BigQuery dataset.
- **POST `/api/v1/access/revoke`**: Revokes previously granted access by removing the IAM binding and updating Firestore.

### 3.4. Chat & Intelligence (Vertex AI)
- **POST `/api/v1/chat`**: Specifically for the "Chat with Table/Dataset" feature. 
  - **How it works**: Uses the Conversational Analytics API or Vertex AI directly, feeding the request along with schema context.

## 4. Understanding Relationship Inference (Dataplex + Gemini)

A frequent question is: **Do we call the Gemini API directly to find table relationships?**
**Answer**: No. We use the **Dataplex DataScans API**, which uses Gemini **internally** on Google's side.

### The Flow:
1. **Triggering the Scan**: If a table has no documented relationships, the backend calls the Dataplex `v1/dataScans` endpoint with a `dataDocumentationSpec`.
   - *Code:* `triggerDocumentationScan` in `services/datasetRelationshipService.js`.
2. **Dataplex + Gemini**: Google Cloud automatically spins up a process, uses Gemini to read the BigQuery schema and data samples, and writes inferred foreign-key relationships into the Dataplex Catalog as an `Aspect`.
3. **Retrieval**: The backend reads this Aspect (via `fetchRelationshipsFromCatalog`), builds a recursive graph (up to 3 levels deep), and caches the result in Firestore for fast frontend loading.

## 5. How to Change or Extend the Backend

1. **Adding a New API Route**:
   - For simple endpoints, add them directly in `server.js` or preferably in the `routes/` directory if the domain is expanding.
   - Example: `app.post('/api/v1/my-new-feature', authMiddleware, async (req, res) => { ... })`

2. **Modifying Access Logic**:
   - If you need to change *how* access is granted (e.g., adding Row-Level security instead of Dataset-level IAM), modify the `grantIamAccess` function in `services/gcpIamService.js`.

3. **Modifying the AI Prompts/Search**:
   - To alter how the AI understands search, update the system instruction string sent to Vertex AI in `server.js` (look for `systemInstruction` near the `/api/v1/ai-search` or `/api/v1/chat` endpoints).

4. **Testing Locally**:
   - Ensure you have a valid `.env` file with `GOOGLE_CLOUD_PROJECT_ID` and GCP ADC (Application Default Credentials) configured.
   - Run `node server.js` or `npm run dev` (if using nodemon).

---

## 6. Known Integrations & Permissions Required

For the backend to function, the Application Default Credentials (or service account running the backend) must have:
- `roles/dataplex.admin` or Viewer (to read Catalog and trigger Scans)
- `roles/bigquery.admin` or DataOwner (to modify IAM policies for access requests)
- `roles/datastore.user` (Firestore read/write)
- `roles/aiplatform.user` (Vertex AI Chat/Search)
